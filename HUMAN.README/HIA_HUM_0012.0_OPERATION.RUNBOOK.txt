01.00_INTRO
==========
Este archivo es el runbook operativo de HIA.
Su objetivo es que un humano (admin/dev/peatón) pueda:
- entender qué archivo editar según el tipo de cambio,
- saber qué script ejecutar,
- saber en qué orden ejecutarlo,
- reconocer el output esperado,
- y saber dónde revisar si algo falla.

REGLA DE AUTORIDAD:
- START.RITUAL manda si hay conflicto con la interacción humano–IA cloud.
- Este archivo manda para operación local y actualización del sistema.
- DragnDrop es GENERATED-ONLY: nunca se edita manualmente.

02.00_PRINCIPIOS_OPERATIVOS
==========
02.10 CANON vs DERIVED
- CANON = HUMAN.README + TOOLS + TRIGGERS.
- DERIVED = DragnDrop\PhaseX\ y otros outputs generados.
- Si un DERIVED está mal: se regenera. No se parchea a mano.

02.20 RESPONSABILIDADES
- HUMAN.README = contratos, reglas, packaging, rituales, troubleshooting.
- 02_TOOLS = implementación real (build/test/sync/validators).
- 05_Triggers = entrypoints simples para el peatón.
- DragnDrop = build artifact para adjuntar a IA cloud.

02.30 NOMENCLATURA
- Usar siempre “Phase” (no “Fase”) en nombres, textos y tooling.

02.40 RADAR EN DRAGNDROP
- IncludeRadar=None por defecto.
- Radar solo por toggle manual:
  - Index
  - IndexLite
- RADAR_FULL (no es output canónico; solo forensics autorizados) no entra por defecto en paquetes DragnDrop.

04.00_SECUENCIA_SEGURA_PARA_ACTUALIZAR_HIA
==========
04.10 Regla general
04.10.01 Editar SOLO el canon que corresponda.
04.10.02 No editar manualmente DragnDrop\PhaseX\.
04.10.03 Después de editar, regenerar y testear.
04.10.04 Recién después hacer prueba en IA cloud.
04.10.05 Recién al final hacer git add / commit / push.

04.20 Orden recomendado de aplicación
04.20.01 Si el cambio afecta DRAGNDROP / PHASES:
1) Editar HUMAN (contract/manifest/ritual) según corresponda.
2) Editar TOOLS si la lógica build/test quedó desalineada.
3) Ejecutar trigger de la phase afectada.
4) Revisar PASS local.
5) Probar en IA cloud.
6) Si todo sale bien: git status / commit / push.

04.20.02 Motivo del orden
- Si cambias tooling sin ajustar HUMAN, el sistema pierde trazabilidad.
- Si cambias HUMAN y no regeneras, DragnDrop queda con drift.
- Si haces commit antes del smoke test, versionas basura.

04.30 DECISION_CANON (MVP1) — Estructura exigida en Phase1
04.30.01 Qué se decidió
- HIA debe exigir definición de ESTRUCTURA INICIAL del proyecto en Phase1 (Project Bootstrap).
- Sin estructura + política de IDs => Phase1 NO se cierra (anti “cierres falsos”).

04.30.02 Por qué
- Si no existe estructura explícita, el peatón improvisa y la IA deriva (carpetas inventadas / nombres inconsistentes / RADAR incoherente).
- El costo de “arreglar después” es mayor que el costo de definir una base mínima en Phase1.

06.00_USO_CON_IA_CLOUD
==========
06.10 Regla de adjuntos
- Adjuntar archivos sueltos de DragnDrop\PhaseX\
- No adjuntar Raw
- No adjuntar 03_ARTIFACTS completos
- No adjuntar ZIP por defecto

06.20 Regla de prompt
- START.RITUAL manda.
- El README del paquete debe coincidir con el contract esperado.
- La IA cloud debe responder primero: "acuso leído"
- Luego debe listar archivos exactos leídos (uno por línea).
- Si falta un archivo requerido: FAIL determinista.
- Si inventa un archivo (adjunto inexistente): FAIL (alucinación).

- Regla anti-alucinación de OUTPUTS (NO negociable):
  - Si la IA declara que “generó/creó/actualizó” un archivo/artefacto/log, DEBE cumplir UNA de estas:
    (1) El archivo está ADJUNTO en el mensaje (cloud), o
    (2) Entrega PATH verificable (REL_PATH + FULL_PATH), o
    (3) Pega el contenido completo inline con nombre exacto del archivo.
  - Si no puede cumplir (1)(2)(3): debe decir “NO SE GENERÓ OUTPUT” y quedarse en PLAN_ONLY.
  - Prohibido referenciar “archivo adjunto” si no existe.

06.30 Modo de operación
- PLAN
- APPLY
- VALIDATE
- LOG

Nota:
- PLAN_ONLY por defecto, salvo que el contract indique APPLY aprobado.

07.00_OUTPUT_ESPERADO_Y_EVIDENCIA
==========
07.10 Output local esperado
- Carpeta DragnDrop\PhaseX\ regenerada.
- README.txt regenerado.
- Tester en PASS.
- Sin edición manual de archivos derivados.

07.20 Evidencia mínima
- output visible del trigger
- output visible del tester
- si aplica: RADAR manual por toggle
- evidencia cloud (si aplica):
  - “acuso leído”
  - lista exacta de archivos leídos (uno por línea)
  - línea de estado: “HIA Fase X activada” o “error codigo FX_FAIL”
  - si la IA declara outputs: evidencia obligatoria (adjunto o REL_PATH+FULL_PATH o inline)
- git status limpio antes de cerrar
- commit/push solo después de PASS

==========
08.00_SI_FALLA
==========
08.10 Primera regla
- No adivinar.
- No editar DragnDrop a mano.
- Copiar error completo.

08.20 Dónde mirar
- terminal actual
- logs del tool ejecutado
- HUMAN.README\10.0_HUMAN.HIA.TROUBLE.txt

08.30 Casos típicos
- falta archivo requerido
- mismatch de nombre canónico
- README sin string literal esperado
- IncludeRadar desalineado
- trigger invoca tool incorrecto
- IA cloud no respeta contract START.RITUAL

08.40 Qué hacer
1) Identificar el primer FAIL real.
2) Corregir canon/tool/trigger según corresponda.
3) Re-ejecutar trigger.
4) Verificar PASS.
5) Recién después repetir prueba cloud.
