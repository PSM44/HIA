==========
02.0_HUMAN.OPERATION — Operación y actualización del sistema (HIA)
==========

02.00 ROL DE ESTE DOCUMENTO (WBS)
02.00.01 Este archivo define PRINCIPIOS operativos y reglas de actualización (académico + conceptual).
02.00.02 Este archivo NO es el runbook de comandos.
02.00.03 El runbook operativo (comandos exactos + orden por Phase + outputs esperados) vive en:
         HUMAN.README\HIA_HUM_0012.0_OPERATION.RUNBOOK.txt
02.00.04 START.RITUAL manda si hay conflicto sobre interacción humano–IA cloud:
         HUMAN.README\09.0_HUMAN.START.RITUAL.txt

02.01 PRINCIPIOS (WBS)
02.01.01 START.RITUAL manda como contrato de interacción humano–IA cloud.
02.01.02 El núcleo permanente de HIA no se define por phases, triggers ni DragnDrop.
02.01.03 Phases y triggers existen como mecanismos de acceso, activación y ejecución peatón-first.
02.01.04 DragnDrop existe como capa de packaging, handoff y fallback para contextos cloud; no como corazón permanente del sistema.
02.01.05 Si algo falla en packaging o activación, no se parchea a mano el derivado: se corrige el canon y se re-ejecuta el trigger o runner correspondiente.
02.01.06 Foco: pocas carpetas, archivos grandes, pero con potencial de escala registrado.
02.02 RUTINA DE EJECUCIÓN — PHASE0 (WBS)
02.02.01 Precondiciones (gates)
02.02.01.01 Existe PROJECT_ROOT (repo HIA).
02.02.01.02 Existe carpeta HUMAN.README con archivos requeridos.
02.02.01.03 Nombres canónicos exactos (no inventar variantes).

02.02.02 RUN — Build + Test (un comando)
02.02.02.01 El comando exacto y el output esperado del trigger viven en el RUNBOOK operativo:
          HUMAN.README\HIA_HUM_0012.0_OPERATION.RUNBOOK.txt
02.02.02.02 Regla:
          - No ejecutar tools sueltos para operación normal.
          - Ejecutar triggers (peatón-first) y dejar que llamen tools internos.

02.02.03 Output esperado (artefacto)
02.02.03.01 Carpeta:
          <PROJECT_ROOT>\DragnDrop\Phase0\
02.02.03.02 Contenido mínimo requerido:
          00.0_HUMAN.GENERAL.txt
          01.0_HUMAN.USER.txt
          07.0_HUMAN.MASTER.txt
          08.0_HUMAN.SYNC.MANIFEST.txt
          09.0_HUMAN.START.RITUAL.txt
          README.txt

02.02.03.03 Contenido condicional (si existe continuidad):
          04.0_HUMAN.BATON.txt

02.02.04 Reintento controlado (si falla)
02.02.04.01 Si falta un archivo -> corregir en HUMAN.README y re-ejecutar trigger.
02.02.04.02 Prohibido: “copiar a mano” dentro de DragnDrop\Phase0.
02.02.04.03 Si el FAIL es por nombre -> normalizar al canon (puntos vs underscore) en HUMAN.README.

02.03 ACTUALIZACIÓN DEL SISTEMA (WBS)
02.03.01 Tipos de cambios
02.03.01.01 Cambio HUMANO (HUMAN): se edita canon (HUMAN.README\*.txt).
02.03.01.02 Cambio IA (TOOLS/TRIGGERS): se edita script en 02_TOOLS / 05_Triggers.
02.03.01.03 Cambio DERIVED: se regenera (no se edita manualmente).

02.03.02 Secuencia segura para cambios en Phase0
02.03.02.01 Editar SOLO canon (HUMAN / TOOLS según corresponda).
02.03.02.02 Ejecutar Build+Test (02.02.02).
02.03.02.03 Probar en IA cloud con prompt endurecido.
02.03.02.04 Recién después: git status / commit / push.

02.04 TROUBLESHOOTING CANÓNICO (WBS)
02.04.01 Caso: “Falta requerido START.RITUAL”
02.04.01.01 Causa típica: mismatch nombre (START_RITUAL vs START.RITUAL).
02.04.01.02 Fix: renombrar a canon + ajustar tooling si corresponde + rerun trigger.
02.04.02 Caso: IA cloud repite datos de Phase0
02.04.02.01 Causa típica: prompt permisivo o IA ignora “PLAN_ONLY”.
02.04.02.02 Fix: usar prompt endurecido (ver 02.05).

02.05 PROMPT CANÓNICO — PHASE0 (WBS)
02.05.01 Autoridad: START.RITUAL manda (contrato humano–IA cloud).
          Archivo: HUMAN.README\09.0_HUMAN.START.RITUAL.txt
02.05.02 Regla: este archivo NO duplica prompts; solo declara el principio de autoridad.
02.05.03 Si el peatón necesita “copy/paste exacto”, usar el RUNBOOK operativo:
          HUMAN.README\HIA_HUM_0012.0_OPERATION.RUNBOOK.txt
