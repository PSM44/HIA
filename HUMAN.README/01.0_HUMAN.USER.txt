00.10_GLOSARIO_MINIMO
==========

TL.Bootstrap (Tooling & Lifecycle Bootstrap): conjunto transversal de métodos/sistemas (RADAR, BATON, CIS, SKILLS, estructura, HUMAN, etc.)
RADAR (Inventario/Auditoría): “foto” del filesystem; niveles LITE/INDEX/CORE (RADAR_FULL solo como excepción forense).
RADAR_LITE (Delta): indica qué cambió desde la última corrida relevante.
RADAR_CORE (Consolidado relevante): texto/código/config relevante para que IA entienda el sistema.
BATON (Estado operativo): continuidad/handoff entre sesiones/chats.
CIS (Content Injection System): contrato explícito de cambios (CREATE/EDIT/MOVE/RENAME/DELETE).
Gate (Aprobación humana): decisión explícita para permitir APPLY.
PLAN_ONLY (Solo plan): modo por defecto; no se toca filesystem.
APPLY (Ejecución): cambios en filesystem (solo con Gate).
VALIDATE (Validación): tests/smoke/runner + verificación outputs.
LOG (Evidencia): registro de OK/FAIL, cambios y next actions.
MB (MiniBattle): unidad mínima de avance verificable.
MVP (Minimum Viable Product): alcance mínimo viable priorizado.
CANON (Cierre oficial): decisión/documento congelado; no se edita directo.
Quarantine (Borrado seguro): zona donde van “deletes” por defecto.

==========

00.25.10_GATES_(MINIMOS)
- Default siempre: PLAN_ONLY.
- Gate humano (APPLY_APPROVED) se usa solo cuando:
  - hay lista exacta de archivos,
  - rollback claro,
  - validación definida,
  - y evidencia esperada.
- Regla de fricción:
  - máximo 1 gate por MiniBattle.
  - si una sesión requiere 5 gates, no es “sesión”: es 5 MiniBattles mal separadas.

00.25.20_DRAG_AND_DROP_(MINIMO)
- Adjuntar archivos es caro (tiempo + ruido + costo de contexto).
- Regla peatón:
  - 0 adjuntos por defecto.
  - 1 adjunto solo si desbloquea la MiniBattle.
  - 2+ adjuntos = requiere justificar “por qué no basta con snippet o path + RADAR_CORE”.

00.25.30_OUTPUTS_A_LA_MANO
- Los archivos “de mando” deben estar siempre a mano (con ubicación explícita):

  - BATON (estado / continuidad):
