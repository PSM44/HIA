01.00_ROL_DEL_HUMANO
==========

01.10_AUTORIDAD_Y_RESPONSABILIDAD
- El humano es dueño de:
  - objetivo,
  - restricciones,
  - gates de ejecución,
  - y promoción a CANON.
- Si el input es ambiguo, el output será inestable (no es culpa de “la IA”, es un problema de definición).

01.20_QUE_NO_SE_DELEGA
- No se delega:
  - aprobación APPLY,
  - decisiones CANON,
  - seguridad,
  - scope,
  - decisión de éxito/fracaso.

01.30_QUE_SI_SE_DELEGA (CON_CONTROL)
- Sí se delega:
  - planes,
  - borradores,
  - scripts sugeridos,
  - análisis de riesgos,
  - revisión de consistencia (RADAR),
  - contratos CIS para aprobar.

02.00_RUTINA_DE_INICIO_DE_PROYECTO
==========

02.10_PREPARACION
- Escribir bullets:
  - objetivo,
  - usuario objetivo,
  - éxito,
  - fracaso,
  - fuera de alcance.

02.20_PRIMERA_INTERACCION_CON_IA (PLAN_ONLY)
- Pedir:
  - plan por fases,
  - riesgos,
  - supuestos,
  - entregables de término por fase.
- Prohibido:
  - “hazlo todo” sin scope,
  - APPLY sin gate.

02.30_SENALES_DE_MALA_DEFINICION
- Plan cambia radicalmente cada iteración.
- No hay DoD.
- Scope se expande sin control.
Corrección:
- volver a Filosofía (Fase 0),
- cerrar definiciones base,
- declarar supuestos.

==========
03.00_RUTINA_DE_CAMBIO_EN_PROYECTO_EXISTENTE
==========

03.10_COMO_PEDIR_CAMBIOS
Siempre incluir:
- objetivo del cambio,
- scope exacto (paths/módulo),
- constraints (no tocar X),
- definición de éxito,
- PLAN_ONLY o APPLY (si se permitirá).

03.20_COMO_APROBAR_APPLY
Antes de aprobar exigir:
- plan paso a paso,
- lista de archivos a tocar,
- rollback (commit checkpoint),
- validación post-change,
- RADAR_LITE pre/post.

03.30_REPLACE_POR_BLOQUES (VS CODE)
- La IA debe entregar:
  (1) BLOQUE_ORIGINAL (título/subtítulo + contenido completo)
  (2) BLOQUE_CORREGIDO (mismo título/subtítulo + contenido completo corregido)
- Tú haces Replace (sin edición manual intermedia).

==========
04.00_RUTINA_DE_VALIDACION_Y_CALIDAD
==========

04.10_VALIDACION_MINIMA
- Tests/lint/build allowlisted si existen.
- Si no: smoke test mínimo (compila/arranca).
- Siempre: revisar logs, outputs y scope.

04.20_ANTE_FAIL
- Regla anti-shotgun:
  - 1 hipótesis,
  - 1 cambio,
  - 1 prueba.
- Si el fallo es tooling:
  - registrar TROUBLE,
  - rollback si contaminó repo.

04.30_EVITAR_FALSOS_OK
- OK solo si:
  - evidencia existe (log/commit/outputs),
  - validación pasó,
  - scope respetado,
  - RADAR post no muestra anomalías.

==========
05.00_RUTINA_DE_CIERRE_DE_SESION
==========

05.10_BATON
Debe quedar:
- objetivo actual,
- decisiones (DRAFT/CANON),
- cambios aplicados (paths),
- validación + evidencia,
- next actions,
- riesgos y supuestos.

05.20_CUANDO_ACTUALIZAR_TL_BOOTSTRAP
- Solo si el cambio es reutilizable como método/política/tool.
- Si fue “solo feature”, NO contamina.

05.30_CUANDO_MIGRAR_DE_CHAT
- Migrar si:
  - repites contexto 3 veces,
  - IA confunde paths o reglas,
  - output inconsistente,
  - costo sube sin progreso.
Antes:
- export/snapshot,
- BATON actualizado,
- RADAR_INDEX + RADAR_LITE (CORE solo si hace falta).
