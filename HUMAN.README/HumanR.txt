01.00_PROCESO_COMPLETO_POR_FASES
==========

01.10_VISTA_POR_FASES
- Fase 0: Filosofía (HUMAN-first)
- Fase 1: Entorno contextual IA (PLAN_ONLY)
- Fase 2: Entorno local (bootstrap)
- Fase 3.1: Pre-Coding (setup check)
- Fase 3.2: Coding (iteración controlada)

01.20_ENTREGABLES_POR_FASE_(DOD)
01.20.01_FASE_0 — FILOSOFIA
- HUMAN del sistema/proyecto:
  - propósito, alcance/no-alcance,
  - principios,
  - riesgos top,
  - definición de éxito,
  - DoD por fase.
Gate de cierre:
- humano confirma intención + DoD.

01.20.02_FASE_1 — CONTEXTO_IA (PLAN_ONLY)
- BATON inicial.
- Workflow adoptado: PLAN→APPLY→VALIDATE→LOG.
- Routing policy del proyecto.
- Políticas: scope/denylist/quarantine/rollback/secrets.
- Decisiones base (qué queda DRAFT y qué se busca llevar a CANON).
Gate de cierre:
- humano aprueba “listo para bootstrap local”.

01.20.03_FASE_2 — LOCAL_BOOTSTRAP
- Estructura local creada (scripts one-shot si aplica).
- Tools allowlisted (RADAR, validadores, sync runner).
- Primer commit Git baseline.
Gate de cierre:
- scripts corren OK + outputs existen + commit baseline.

01.20.04_FASE_3_1 — PRE_CODING
- Setup check:
  - comando run,
  - dependencias instalables,
  - smoke test o tests mínimos,
  - logs.
Gate de cierre:
- validate OK con evidencia.

01.20.05_FASE_3_2 — CODING
- Iteración por cambios:
  - PLAN (obligatorio),
  - APPLY (solo con gate),
  - VALIDATE,
  - LOG,
  - commit.
Gate de cierre por feature:
- DoD del feature + validate OK + commit.

01.30_GATES_(REGLA_GENERAL)
- Una fase se cierra cuando:
  - entregables existen,
  - criterios cumplidos,
  - decisión registrada en BATON,
  - checkpoint/commit cuando corresponde.

01.40_ARTEFACTOS_MINIMOS_Y_UBICACION
- HUMAN.README: documentos HUMAN (General/User/PF0/Baton/Radar/CIS/Master)
- 02_TOOLS: scripts allowlisted
- 03_ARTIFACTS: logs/snapshots/quarantine/historial
- Raw: fuentes externas (PDFs), fuera de CORE/LITE por P0

02.00_ROUTING_OPERATIVO
==========

02.10_MINIMO_CONTEXTO_EFECTIVO
- Resolver con el mínimo contexto posible.
- Escalar solo cuando el tipo de tarea lo exige.

02.20_TAXONOMIA_(ABREVIACIONES_EXPLICADAS)
TaskType: tipo de tarea (plan, doc, code, debug, audit).
ContextLevel:
- L0: mínimo
- L1: snippets selectivos
- L2: CORE por módulo/APP_SCOPE
- L3: FULL puntual
ApplyGate:
- PLAN_ONLY
- APPLY_APPROVED

02.30_PLANTILLA_MATRIZ_ROUTING
Por cada TaskType:
- ContextLevel recomendado
- Output mínimo requerido
- Gate permitido
- Evidencia requerida

02.40_ESCALAMIENTO
- Subir a L2 si:
  - refactor multiarchivo,
  - debug complejo,
  - o falló por falta de contexto.
- L3 solo si:
  - auditoría puntual,
  - incidente severo,
  - forensic ordenado por humano.

==========

03.10_REGLAS
- Pre APPLY: RADAR_LITE
- Post APPLY: RADAR_LITE
- Navegación: RADAR_INDEX
- Refactor/debug: RADAR_CORE por módulo/APP_SCOPE

03.20_EXCLUSIONES_P0_HIA
- CORE/LITE NO consideran:
  - Raw
  - 03_ARTIFACTS (logs/old/deadhistory)
- Implementación recomendada: excluir \Raw\ y \03_ARTIFACTS\ completos.

03.30_SENALES_SALUD
- created/edited/moved/deleted
- anomalías naming/IDs
- out-of-scope writes

03.40_INCIDENTES
- Capturar INDEX + LITE inmediato.
- CORE por módulo si se requiere.
- FULL solo orden humana.
