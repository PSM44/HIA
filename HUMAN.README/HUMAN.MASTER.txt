Single-Edit Rule (Regla de edición única): cada regla se edita en un solo lugar canónico.
FULL-copy (Copia completa): derivación que replica texto completo (no resumen), usada en HUMAN por requerimiento académico.
Gate (Aprobación humana): permiso explícito para APPLY.

01.00_PROPOSITO_Y_MENTAL_MODEL
==========

01.10_PROPOSITO
- Este archivo existe por una razón práctica:
  cuando tienes muchos archivos HUMAN, cualquier cambio se vuelve engorroso si exige editar 10 copias divergentes.
- MASTER define un sistema para:
  - mantener ultra detalle en HUMAN,
  - sin edición manual multiarchivo,
  - con una única fuente de verdad por regla,
  - y sincronización automática FULL-copy.

01.20_MENTAL_MODEL
- Cada dominio de reglas tiene un “archivo dueño”.
- Ese archivo dueño contiene el CANON.
- Otros archivos pueden incluir el mismo contenido como DERIVED (FULL-copy) por usabilidad.
- Los derivados se actualizan por SYNC.RUNNER usando SYNC.MANIFEST.
- Git + RADAR + BATON entregan evidencia y trazabilidad.

02.00_PRINCIPIOS_META_(REGLAS_DE_ORO)
==========

02.10_SINGLE-EDIT_RULE (Regla de edición única)
- Una regla o plantilla se edita en un solo lugar canónico (archivo dueño del dominio).

- Si hay conflicto: manda el archivo dueño del dominio.

06.50.30_RADAR_NIVELES_(DERIVED)
<<<DERIVED_BEGIN ID=DERIVED.RADAR.LEVELS.MASTER.0001 SOURCE=HUMAN.RADAR.0001::02.00 MODE=FULL>>>
02.10_RADAR_LITE_(DELTA_OPERATIVO)
Objetivo:
- detectar cambios rápidamente con bajo costo.

Salida mínima:
- lista de archivos creados/editar/eliminados/movidos (según detección)
- rutas + timestamps + tamaño (mínimo)
- hash para texto si aplica y es costo-efectivo

Uso:
- antes y después de APPLY (mínimo obligatorio).

02.20_RADAR_INDEX_(FOTO_INVENTARIO)
Objetivo:
- inventario navegable del repo/proyecto.

Salida mínima:
- tree/listado de rutas relevantes
- metadatos (size, modified, created)
- hash opcional para texto según política

Uso:
- navegación,
- handoff,
- revisión de estructura.

02.30_RADAR_CORE_(FOTO_CONTENIDO_RELEVANTE_PARA_IA)
Objetivo:
- consolidar el contenido legible RELEVANTE (código+config+tests según política),
  para que una IA “sepa el sistema” y pueda:
  - detectar inconsistencias,
  - corroborar coherencia,
  - ubicar responsabilidades,
  - apoyar refactor/debug (siempre con validate externo).

Reglas:
- CORE NO incluye binarios.
- CORE se ejecuta por scope:
  - CORE_SCOPE=APP (recomendado default)
  - CORE_SCOPE=MODULE:<x> para refactor puntual
  - CORE_SCOPE=REPO solo si es razonable y con exclusiones fuertes
- CORE tiene size limits por archivo y total consolidado (para evitar colapsar).

02.40_RADAR_FULL_(FOTO_AUDITORIA_COMPLETA)
Objetivo:
- auditoría completa (alto costo, alto ruido) para:
  - incidentes severos,
  - auditorías manuales,
  - preparación de export/handoff “forensic”.

Reglas:
- FULL solo bajo orden explícita del humano (no default).
- FULL requiere exclusiones estrictas, o el sistema deja de ser útil.

08.00_POLITICA_DE_CONFLICTOS
==========

08.10_DERIVED_EDITADO_MANUALMENTE
- Si un derivado fue editado manualmente:
  - se considera DIRTY.
  - se debe resolver:
    A) re-sincronizar desde fuente (preferido), o
    B) promover el cambio a la fuente canónica y luego sync, o
    C) marcar override (solo excepcional).

08.20_CANON_CONFLICTO
- Si dos fuentes canónicas contradicen:
  - manda el archivo dueño del dominio (DOC.AUTHORITY.MAP).
  - si el conflicto es meta, manda MASTER.
