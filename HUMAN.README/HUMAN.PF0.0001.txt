00.10_GLOSARIO_MINIMO_(ABREVIACIONES_EXPLICADAS)
==========

HIA (Human + IA): entorno y metodología para colaborar Humano e Inteligencia Artificial en desarrollo asistido de código.
TL.Bootstrap (Tooling & Lifecycle Bootstrap): conjunto transversal de componentes reutilizables (RADAR, BATON, CIS, SKILLS, estructura, HUMAN, etc.).
PF.0 (PFundations Level 0): método mínimo operativo (esta especificación) para operar HIA sin fricción.
Ritual de sesión: protocolo repetible de inicio y cierre de una sesión de trabajo.
MiniBattle (Minibattle, MB): unidad mínima de avance verificable, con output, validación y evidencia.
MVP (Minimum Viable Product): alcance mínimo viable; aquí se usa como MVP del sistema HIA.
Definition of Done (Definición de Terminado, DoD): criterios explícitos para declarar una fase o minibattle como “terminada”.
Gate (Aprobación humana): decisión explícita del humano para permitir APPLY (ejecución) vs PLAN_ONLY (solo planificación).
PLAN_ONLY (Solo plan): modo por defecto; no se modifica filesystem.
APPLY (Aplicar): modo de ejecución; se modifica filesystem (solo con gate).
VALIDATE (Validación): prueba mínima (tests/lint/build o smoke test) + verificación de outputs.
LOG (Evidencia/Registro): registro reproducible de OK/FAIL, cambios y next actions.
CANON (Cierre oficial): estado en que un asunto/documento queda cerrado y congelado; se promueve solo por humano.
RADAR (Inventario/Auditoría): “foto” del filesystem y delta (RADAR_LITE).
RADAR_LITE (Delta): lista de cambios desde corrida anterior relevante.
RADAR_CORE (Consolidado relevante): contenido legible relevante para que IA entienda el sistema (definido por APP_SCOPE).
BATON (Estado operativo): continuidad/handoff; resume objetivo, decisiones, cambios y evidencia.
CIS (Content Injection System): contrato explícito de cambios (CREATE/EDIT/MOVE/RENAME/DELETE); separación Planner/Executor.
APP_SCOPE (Application Scope): definición operativa de qué carpetas/archivos conforman “la aplicación” para RADAR_CORE; se define por proyecto y se mantiene actualizado.

01.00_PROPOSITO_Y_MENTAL_MODEL
==========

01.10_PROPOSITO
- PF.0 existe para resolver un problema concreto:
  “control fuerte” suele volverse lento porque cada cambio exige demasiados pasos manuales y produce propagación multiarchivo.
- PF.0 define un método mínimo que:
  - preserva gobernanza (control),
  - maximiza throughput (avance real),
  - y reduce fricción (menos trabajo manual repetitivo).

01.20_MENTAL_MODEL
- Una sesión de trabajo es una secuencia de MiniBattles (Minibattles).
- Cada MiniBattle debe producir un output verificable y evidencia.
- El sistema se sostiene con:
  - gates claros (PLAN_ONLY por defecto),
  - evidencia reproducible,
  - y continuidad (BATON) entre sesiones.

01.30_LO_QUE_PF0_NO_PERMITE
- No permite ejecución sin gate.
- No permite “OK” sin evidencia.
- No permite cambios fuera de scope (PROJECT_ROOT por defecto).
- No permite “borrar” sin quarantine por defecto.

==========
02.00_PRINCIPIOS_OPERATIVOS_PF0
==========

02.10_MINIMO_NO_NEGOCIABLE
- PLAN → APPLY → VALIDATE → LOG (siempre).
- Gate humano explícito para APPLY.
- Git como rollback primario.
- Delete=quarantine por defecto.
- RADAR_LITE antes/después de APPLY (delta para confirmar cambios).
- BATON actualizado al cierre.

02.20_REGLA_DE_FRICCION_MINIMA
- El método no debe convertirse en ceremonia infinita.
- Por defecto:
  - 1–3 MiniBattles por sesión
  - 1 Gate máximo por minibattle (si aplica)
  - 1 validación mínima por minibattle

02.30_REGLA_DE_CAMBIO_UNICO_REAL
- Para evitar “propagación manual”:
  - una regla se edita en un solo lugar canónico (archivo dueño),
  - y se replica FULL (texto completo) por SYNC donde sea necesario.
- PF.0 define el método; otros archivos pueden incluirlo por derivación FULL.

==========
03.00_ESTRUCTURA_DEL_TRABAJO
==========

03.10_UNIDAD_DE_TRABAJO: MINIBATTLE
- La unidad operativa no es “una sesión” ni “un feature”.
- Es una MiniBattle (Minibattle): output verificable + evidencia.

03.20_SESION
- Una sesión consiste en:
  - Start Ritual (inicio)
  - 1–3 MiniBattles
  - End Ritual (cierre)

03.30_EVIDENCIA
- Evidencia mínima aceptable:
  - log de validación OK/FAIL,
  - commit Git (cuando hubo APPLY),
  - delta RADAR_LITE pre/post APPLY,
  - update BATON (estado y next actions).

==========
04.00_RITUALES_DE_SESION
==========

04.10_RITUAL_DE_INICIO_(START_RITUAL)
Objetivo:
- arrancar sin re-explicar el universo y sin drift.

Pasos:
04.10.01 Leer BATON
- revisar objetivo actual, decisiones (DRAFT/CANON), bloqueos, next actions.

04.10.02 Seleccionar MiniBattles
- elegir 1–3 MiniBattles con outputs claros.

04.10.03 Definir modo (Gate)
- Default: PLAN_ONLY.
- Si una MiniBattle requiere APPLY:
  - exigir plan, lista exacta de archivos, rollback, validación.

04.10.04 Confirmar Scope
- declarar qué paths se pueden tocar.
- confirmar que PROJECT_ROOT es frontera por defecto.

04.10.05 Confirmar APP_SCOPE (si hay RADAR_CORE)
- si la MiniBattle implica “comprensión total de aplicación”:
  - confirmar APP_SCOPE actualizado (por proyecto) para que CORE tome solo lo relevante.

04.20_RITUAL_DE_CIERRE_(END_RITUAL)
Objetivo:
- continuidad y trazabilidad.

Pasos:
04.20.01 Actualizar BATON
- objetivo actual, decisiones, cambios aplicados, evidencia, next actions.

04.20.02 Evidencia mínima
- adjuntar/registrar logs y commits.
- confirmar RADAR_LITE post-APPLY si hubo cambios.

04.20.03 Decidir TL.Bootstrap update
- solo si:
  - descubriste patrón reusable,
  - cambiaste método,
  - o mejoraste control/seguridad transversal.
- Si fue solo trabajo del proyecto, NO contamina TL.Bootstrap.

04.20.04 Preparar handoff si aplica
- si se migrará chat/IA:
  - BATON + RADAR_INDEX + RADAR_LITE (+ RADAR_CORE si necesario).

05.00_MINIBATTLES_(MB)
==========

05.10_DEFINICION
- MiniBattle (Minibattle, MB) = unidad mínima de avance verificable.
- Debe terminar en:
  - DONE (terminada) o
  - BLOCKED (bloqueada con causa explícita y next action).

05.20_COMPONENTES_OBLIGATORIOS_DE_UNA_MB
- MB_ID (ID único WBS-friendly)
- Objetivo (qué cambia en el mundo)
- Inputs requeridos (qué necesito)
- Plan (pasos)
- Output (artefacto verificable)
- Gate requerido (PLAN_ONLY o APPLY_APPROVED)
- Validación (qué prueba lo confirma)
- Evidencia (log/commit/radar)
- Estado (OPEN/DONE/BLOCKED)
- Riesgo principal y mitigación (1 línea mínima)

05.30_TIPOS_DE_MB (para routing)
- MB.DOC: documentación HUMAN/TL.Bootstrap
- MB.TOOL: tooling/scripts/validadores
- MB.CODE: código de aplicación
- MB.DEBUG: troubleshooting
- MB.AUDIT: auditoría (RADAR_CORE/FULL, etc.)

05.40_CIERRE_DE_MB (DoD de MB)
- DONE si:
  - output existe,
  - validación OK o criterio equivalente,
  - evidencia registrada,
  - next action actualizada si corresponde.

- BLOCKED si:
  - falta dato duro,
  - falta decisión humana,
  - o hay fallo técnico sin hipótesis clara.
  Debe registrar:
  - causa,
  - plan para desbloquear,
  - next action.

06.00_MVP_DEL_SISTEMA_HIA_(MVP.HIA)
==========

06.10_PROPOSITO
- MVP aquí no es una app: es la capacidad mínima del sistema HIA para operar proyectos.

06.20_MVP.HIA_P0 (must-have)
- Operar PLAN→APPLY→VALIDATE→LOG con:
  - Git (rollback),
  - RADAR_LITE pre/post APPLY,
  - BATON al cierre,

  - contratos de cambio explícitos,
  - separación Planner/Executor.
- Routing policy:
  - TaskType → ContextLevel → Gate.

06.40_MVP.HIA_P2 (could-have)
- Sync system (CANON.REGISTRY + SYNC MANIFEST + RUNNER) para derivar secciones FULL y evitar propagación manual.
- Quality gates avanzados (linters/CI) por StackProfile (perfil de stack) cuando exista.

08.00_DECISIONES_DRAFT_VS_CANON
==========

08.10_DRAFT
- decisión en exploración; puede cambiar.
- debe registrar:
  - supuestos,
  - por qué,
  - evidencia parcial.

08.20_CANON
- decisión cerrada oficialmente por humano.
- requiere:
  - cabos sueltos resueltos,
  - validación mínima,
  - evidencia,
  - congelamiento (no editar directo; hotfix por branch).

==========
09.00_INTEGRACION_CON_RADAR_BATON_CIS
==========

09.10_RADAR (foto/delta)
- PF.0 exige:
  - RADAR_LITE pre/post APPLY (delta).
- RADAR_CORE se usa cuando:
  - la IA necesita “comprensión completa de la aplicación”,
  - con APP_SCOPE actualizado por proyecto.

09.20_BATON (continuidad)
- PF.0 exige:
  - BATON actualizado al cierre,
  - y antes/después de APPLY significativo si se requiere trazabilidad.

09.30_CIS (contratos de cambio)
- PF.0 no redefine CIS.
- PF.0 exige que cambios aplicados estén declarados de forma explícita:
  - operaciones claras,
  - scope claro,
  - evidencia clara.

10.00_STACK_DISCOVERY_BY_ITERATION
==========

10.10_PRINCIPIO
- No se define stack “por religión”.
- Se define por:
  - requisitos del proyecto,
  - constraints,
  - y evidencia (smoke tests, complejidad, costos, riesgos).

10.20_INSTANCIA_DE_ITERACION (ARTEFACTO)
- Por proyecto, la IA debe crear y mantener:
  STACK.ITERATION.LOG.txt (nombre referencial)
con:
  - hipótesis de stack/setup,
  - comandos run/test,
  - resultados de validación,
  - decisiones DRAFT/CANON,
  - riesgos y mitigaciones.

10.30_CIERRE
- Cuando el stack/setup queda estable:
  - promover decisión a CANON (humano),
  - registrar en BATON,
  - actualizar APP_SCOPE y validadores.

12.00_CONTROLES_DE_FRICCION
==========

12.10_REGLA_80_20
- No perfeccionar 20 documentos por un cambio menor.
- Usar:
  - single-edit canon,
  - derivación FULL por SYNC,
  - y runner para actualizar derivados.

12.20_SINCRONIZACION_(DERIVED_FULL)
- Secciones duplicadas en HUMAN se mantienen por SYNC (no a mano).
- Esto preserva “ultra detallado” sin “trabajo manual infernal”.

12.30_MAXIMO_DE_PASOS_MANUALES_POR_CAMBIO
- Por defecto:
  - 1 edición canónica
  - 1 ejecución runner (sync)
  - 1 validación mínima
  - 1 update baton
  - 1 commit
Si supera eso sistemáticamente, el sistema está mal diseñado.
