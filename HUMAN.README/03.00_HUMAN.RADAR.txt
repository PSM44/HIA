00.20_PROPOSITO_Y_MENTAL_MODEL
==========

00.21_MENTAL_MODEL_SIMPLE
- RADAR es un sistema para responder 3 preguntas:
  (1) ¿Qué existe? (inventario)
  (2) ¿Qué cambió? (delta)
  (3) ¿Qué es relevante? (scope e inclusión/exclusión)

00.22_DOS_SALIDAS_DISTINTAS
- FOTO: muestra el estado actual (snapshot).
- DELTA: muestra cambios desde la última “foto” relevante.

00.23_POR_QUE_IMPORTA
- Sin inventario: no hay navegación ni control.
- Sin delta: no hay disciplina (drift).
- Sin relevancia: RADAR se convierte en ruido y deja de servir.

==========
00.30_DEFINICIONES_BASE
==========

00.31_FOTO_(SNAPSHOT)
- Estado actual del filesystem (FS) dentro de un scope definido.

00.32_DELTA
- Comparación entre “estado anterior” y “estado actual” para listar cambios.

00.33_RELEVANTE
- “Relevante” = útil para entender, mantener y validar la aplicación:
  - código,
  - configuración,
  - infraestructura declarativa (si aplica),
  - tests,
  - scripts del proyecto necesarios para operar.

00.34_IRRELEVANTE
- “Irrelevante” = alto volumen o bajo valor para entender el sistema:
  - dependencias masivas,
  - caches,
  - builds/outputs,
  - logs,
  - artefactos generados automáticamente,
  - binarios no interpretables.

==========
01.00_QUE_ES_RADAR_(Y_QUE_NO_ES)
==========

01.10_RADAR_COMO_FOTO
- RADAR_INDEX / RADAR_CORE / RADAR_FULL generan “fotos” (snapshots).
- Una foto depende de:
  - scope,
  - política de relevancia,
  - exclusiones e inclusiones.

01.20_RADAR_COMO_DELTA (RADAR_LITE)
- RADAR_LITE NO es “foto completa”.
- RADAR_LITE responde: “qué cambió” desde la corrida anterior.
- Es el “instrumento diario” para validar que el APPLY hizo lo que prometía.

01.30_RADAR_NO_REEMPLAZA_TESTS
- RADAR no prueba semántica del software.
- RADAR prueba trazabilidad del filesystem:
  - qué se tocó,
  - qué se movió,
  - qué se eliminó,
  - qué se creó,
  - si hay anomalías de estructura.

==========
02.00_NIVELES_RADAR
==========

02.10_RADAR_LITE_(DELTA_OPERATIVO)
Objetivo:
- detectar cambios rápidamente con bajo costo.

Salida mínima:
- lista de archivos creados/editar/eliminados/movidos (según detección)
- rutas + timestamps + tamaño (mínimo)
- hash para texto si aplica y es costo-efectivo

Uso:
- antes y después de APPLY (mínimo obligatorio).

02.20_RADAR_INDEX_(FOTO_INVENTARIO)
Objetivo:
- inventario navegable del repo/proyecto.

Salida mínima:
- tree/listado de rutas relevantes
- metadatos (size, modified, created)
- hash opcional para texto según política

Uso:
- navegación,
- handoff,
- revisión de estructura.

02.30_RADAR_CORE_(FOTO_CONTENIDO_RELEVANTE_PARA_IA)
Objetivo:
- consolidar el contenido legible RELEVANTE (código+config+tests según política),
  para que una IA “sepa el sistema” y pueda:
  - detectar inconsistencias,
  - corroborar coherencia,
  - ubicar responsabilidades,
  - apoyar refactor/debug (siempre con validate externo).

Reglas:
- CORE NO incluye binarios.
- CORE se ejecuta por scope:
  - CORE_SCOPE=APP (recomendado default)
  - CORE_SCOPE=MODULE:<x> para refactor puntual
  - CORE_SCOPE=REPO solo si es razonable y con exclusiones fuertes
- CORE tiene size limits por archivo y total consolidado (para evitar colapsar).

02.40_RADAR_FULL_(FOTO_AUDITORIA_COMPLETA)
Objetivo:
- auditoría completa (alto costo, alto ruido) para:
  - incidentes severos,
  - auditorías manuales,
  - preparación de export/handoff “forensic”.

Reglas:
- FULL solo bajo orden explícita del humano (no default).
- FULL requiere exclusiones estrictas, o el sistema deja de ser útil.

==========
03.00_RELEVANCIA_Y_EXCLUSIONES
==========

03.10_EL_PROBLEMA_REAL
- Si CORE/FULL incluyen todo lo legible, se pudre:
  - logs, dumps, outputs, caches => ruido
  - la IA aprende basura y se confunde
  - suben costos y tiempos

03.20_CORE_SCOPE (CANON)
- CORE_SCOPE=APP:
  - solo carpetas de aplicación (ej. src/, app/, server/, config/, tests/)
- CORE_SCOPE=MODULE:<x>:
  - solo el módulo objetivo (refactor/debug)
- CORE_SCOPE=REPO:
  - todo el repo con exclusiones fuertes (solo si el repo es “limpio”)

03.30_INCLUDELIST_Y_EXCLUDELIST
- INCLUDELIST define el lente (qué sí importa).
- EXCLUDELIST define el filtro (qué nunca importa).
- Estas listas deben existir por stack (Node/Python/.NET/etc.) y por proyecto.

03.40_BINARIOS_Y_LIMITES
- Binarios: se indexan como metadata (nombre/size/timestamp) pero no se consolidan.
- Límite por archivo:
  - evita consolidar artefactos gigantes o minificados.
- Límite total:
  - evita outputs imposibles de consumir por IA.

==========
04.00_SENALES_Y_OUTPUTS
==========

04.10_SENALES_MINIMAS
- created_files
- edited_files
- moved_or_renamed_files (si se detecta)
- deleted_files (con referencia a quarantine si aplica)

04.20_ANOMALIAS
- naming fuera de estándar WBS/ID
- duplicados funcionales
- archivos fuera de scope o fuera del root
- canónicos sin metadatos/índice

04.30_EVIDENCIA
- logs con OK/FAIL
- hashes (texto) preferidos vs timestamps (más débiles)
- commits Git como evidencia de cambios

==========
05.00_EJECUCION
==========

05.10_MINIMO_OBLIGATORIO
- Antes de APPLY: RADAR_LITE
- Después de APPLY: RADAR_LITE

05.20_NAVEGACION_Y_HANDOFF
- Para ubicar cosas: RADAR_INDEX
- Para “alimentar IA” con código/config: RADAR_CORE (por APP o módulo)

05.30_INCIDENTES_Y_AUDITORIA
- Full solo bajo orden humana.
- Core por módulo suele ser suficiente para debug/refactor.

==========
06.00_INTEGRACION_CON_GIT_CANON_QUARANTINE
==========

06.10_GIT_CHECKPOINTS
- Antes de APPLY significativo: commit checkpoint
- Después de éxito: commit de cambio
- RADAR_LITE ayuda a verificar que lo tocado coincide con lo planeado.

06.20_DELETE_QUARANTINE
- Delete por defecto => mover a quarantine.
- RADAR debe registrar esa “eliminación” como moved/deleted + destino quarantine.

06.30_CANON
- RADAR ayuda a congelar:
  - detecta drift,
  - detecta modificaciones inesperadas en canónicos,
  - facilita auditoría previa a promoción CANON.
