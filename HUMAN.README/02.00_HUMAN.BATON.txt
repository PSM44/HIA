00.20_PROPOSITO_Y_MENTAL_MODEL
==========

00.21_MENTAL_MODEL_SIMPLE
- BATON es el “estado operativo” del proyecto:
  si hoy cierras, mañana puedes continuar en otro chat o con otra IA sin perder control.

00.22_POR_QUE_EXISTE
- Los chats se degradan (performance, drift, confusión).
- La continuidad no puede depender de memoria o de “historial del chat”.
- BATON es el puente: handoff determinista.

==========
01.00_QUE_ES_BATON_(Y_QUE_NO_ES)
==========

01.10_BATON_NO_ES_HUMAN
- HUMAN = filosofía, principios, intención, y definiciones canónicas conceptuales.
- BATON = estado operativo: qué está ocurriendo y qué falta.

01.20_BATON_ES_CONTINUIDAD
- BATON debe permitir:
  - retomar trabajo sin re-explicar todo,
  - identificar decisiones y pendientes,
  - evitar repetir errores,
  - ubicar evidencias (logs, commits, RADAR outputs).

==========
02.00_ESTRUCTURA_MINIMA_DE_BATON_(PLANTILLA)
==========

02.10_CURRENT_OBJECTIVE
- Qué objetivo se está atacando ahora.
- Debe ser concreto, medible y acotado.

02.20_STATE_SUMMARY
- Hecho:
  - hitos completados
- En progreso:
  - piezas activas
- Bloqueos:
  - qué impide avanzar (datos faltantes, dependencias, decisiones)

02.30_DECISIONS
Por cada decisión:
- ID (WBS/único)
- Qué se decidió (texto corto)
- Por qué (razón)
- Estado:
  - DRAFT (aún discutible)
  - CANON (cerrado oficialmente)
- Evidencia:
  - link a archivo, commit, o sección

02.40_CHANGES_APPLIED
- Lista de cambios aplicados en filesystem:
  - operación (CREATE/EDIT/MOVE/RENAME/DELETE)
  - paths
  - commit id (si aplica)
  - quarantine path (si aplica)

02.50_VALIDATION_EVIDENCE
- OK/FAIL + evidencia:
  - logs
  - outputs esperados
  - comandos ejecutados
- Sin evidencia no hay “OK”.

02.60_RISKS_AND_ASSUMPTIONS
- Riesgos abiertos (top 5–10)
- Supuestos activos (lo que se asumió por falta de datos)
- Plan para validar supuestos (cómo se obtiene dato duro)

02.70_NEXT_ACTIONS
- Lista WBS:
  - prioridad (P0/P1/P2)
  - owner (humano/IA planner/IA executor)
  - prerequisitos
  - definición de terminado de cada acción

02.80_HANDOFF_PACKAGE
- Al migrar chat/IA:
  - BATON actualizado
  - RADAR_INDEX + RADAR_LITE recientes
  - Si se necesita “IA omnisciente”: RADAR_CORE por APP o módulo
  - Snapshot/export (si aplica)
  - Último commit id

03.00_CUANDO_ACTUALIZAR_BATON
==========

03.10_AL_CIERRE_DE_CADA_SESION (OBLIGATORIO)
- mínimo:
  objective + decisions + changes + validation + next actions

03.20_ANTES_Y_DESPUES_DE_APPLY_SIGNIFICATIVO
- Antes:
  - registrar intención y scope
- Después:
  - registrar resultados y evidencia

03.30_ANTES_DE_MIGRAR_CHAT
- Si migras sin baton, pierdes continuidad (y repites trabajo).

==========
04.00_BATON_Y_CONTROL_DE_CAMBIO
==========

04.10_BATON_CON_GIT
- Git provee rollback.
- BATON apunta a commits como evidencia.

04.20_BATON_CON_RADAR
- RADAR_INDEX da mapa del repo.
- RADAR_LITE da delta de cambios.
- RADAR_CORE alimenta IA para comprensión del sistema (si se requiere).

04.30_BATON_CON_CIS
- BATON debe referenciar contratos CIS ejecutados:
  - qué se ejecutó,
  - qué se validó,
  - dónde está la evidencia.

==========
05.00_MIGRACION_DE_CHAT_IA_EN_BLANCO
==========

05.10_SENALES_DE_DRIFT
- repetir contexto varias veces
- outputs inconsistentes
- confusión de paths o reglas
- “olvidos” operativos dentro de la sesión
- costo/tiempo sube sin progreso

05.20_PROCEDIMIENTO_DE_MIGRACION
Paso 1: Actualizar BATON (estado completo)
Paso 2: Ejecutar RADAR_INDEX + RADAR_LITE (foto + delta)
Paso 3: Si es necesario, RADAR_CORE por APP o módulo
Paso 4: Crear snapshot/export (si se requiere)
Paso 5: Abrir nuevo chat y “inyectar” solo:
        - BATON
        - RADAR_INDEX (+ LITE)
        - CORE solo si la tarea lo exige

05.30_QUE_EVITAR
- no pegar “todo el repo” sin necesidad
- no re-inyectar FULL como default
- no duplicar logs y basura (mantener relevancia)
