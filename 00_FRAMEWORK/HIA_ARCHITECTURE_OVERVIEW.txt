00.20_PROPOSITO_DEL_SISTEMA
======================================================================

HIA (Human-Integrated Automation) es un framework para coordinar
trabajo humano y trabajo de IA dentro de un proyecto estructurado.

El sistema busca:

- mantener gobernanza humana
- permitir ejecución asistida por IA
- generar evidencia reproducible
- preservar contexto de largo plazo

HIA no es un agente autónomo.
Es un sistema de coordinación humano + IA.

======================================================================
01.00_PRINCIPIOS_DE_DISENO
======================================================================

01.01 HUMAN_FIRST

El humano define intención, reglas y alcance.

Solo HUMAN puede editar manualmente los archivos
dentro de HUMAN.README.

El resto del sistema deriva de esa fuente.

------------------------------------------------

01.02 TXT_FIRST

Todos los artefactos críticos del sistema son texto.

Motivos:

- auditabilidad
- compatibilidad universal
- facilidad de versionamiento en Git
- lectura por humanos e IA

------------------------------------------------

01.03 EVIDENCIA_OBLIGATORIA

Toda ejecución debe generar evidencia mínima.

Formato mínimo:

- resultado OK / FAIL
- log reproducible
- referencia de archivos afectados

------------------------------------------------

01.04 REVERSIBILIDAD

Toda modificación del sistema debe poder revertirse.

Rollback primario:

Git

Rollback secundario:

artifacts y logs

------------------------------------------------

01.05 SCOPE_LIMITADO

Las IA operan únicamente dentro del PROJECT_ROOT.

Se prohíbe ejecución arbitraria fuera del repositorio.

03.00_KERNEL_HIA
======================================================================

El Kernel es el núcleo mínimo universal del sistema.

Su función es coordinar:

- método de trabajo
- roles
- evidencia
- políticas
- identidad

El Kernel debe ser pequeño y estable.

Componentes del Kernel:

------------------------------------------------

03.10 METHOD_KERNEL

Flujo obligatorio de trabajo:

PLAN → APPLY → VALIDATE → LOG

PLAN_ONLY es el modo por defecto.

APPLY requiere gate humano.

------------------------------------------------

03.20 ROLES_KERNEL

Roles definidos:

HUMANO
IA_PLANNER
IA_EXECUTOR

HUMANO:
define intención y aprueba cambios.

IA_PLANNER:
analiza, propone arquitectura o tareas.

IA_EXECUTOR:
implementa código o ejecuta herramientas.

------------------------------------------------

03.30 EVIDENCE_KERNEL

Todo resultado debe producir evidencia mínima.

Formato:

OK / FAIL

+ log
+ paths
+ commit si aplica

------------------------------------------------

03.40 POLICY_KERNEL

Define restricciones de ejecución.

Incluye:

scope boundaries
quarantine delete
rollback obligatorio
gates humanos

------------------------------------------------

03.50 IDENTITY_KERNEL

Define disciplina estructural.

Incluye:

ID_UNICO
VERSION
FECHA
TZ
WBS

05.00_FLUJO_OPERATIVO
======================================================================

Ejemplo de ejecución.

1. HUMANO define tarea.

2. IA_PLANNER analiza problema.

3. PLAN es generado.

4. HUMANO aprueba APPLY.

5. IA_EXECUTOR ejecuta herramientas.

6. VALIDATE verifica resultado.

7. LOG registra evidencia.
