00.20_PROPOSITO_Y_MENTAL_MODEL


==========





00.21_MENTAL_MODEL_SIMPLE


- BATON es el “estado operativo” del proyecto:


  si hoy cierras, mañana puedes continuar en otro chat o con otra IA sin perder control.





00.22_POR_QUE_EXISTE


- Los chats se degradan (performance, drift, confusión).


- La continuidad no puede depender de memoria o de “historial del chat”.


- BATON es el puente: handoff determinista.





00.23_REGLA_DE_ORO


- Si BATON está malo o incompleto:


  mañana repites trabajo, o peor: aplicas cambios con contexto falso.





==========


01.00_QUE_ES_BATON_(Y_QUE_NO_ES)


==========





01.10_BATON_NO_ES_HUMAN


- HUMAN = filosofía, principios, intención, y definiciones canónicas conceptuales.


- BATON = estado operativo: qué está ocurriendo y qué falta.





01.20_BATON_ES_CONTINUIDAD


- BATON debe permitir:


  - retomar trabajo sin re-explicar todo,


  - identificar decisiones y pendientes,


  - evitar repetir errores,


  - ubicar evidencias (logs, commits, RADAR outputs).





==========


02.00_ESTRUCTURA_MINIMA_DE_BATON_(PLANTILLA)


==========





02.10_CURRENT_OBJECTIVE


- Qué objetivo se está atacando ahora.


- Debe ser concreto, medible y acotado.





02.20_STATE_SUMMARY


- Hecho:


  - hitos completados


- En progreso:


  - piezas activas


- Bloqueos:


  - qué impide avanzar (datos faltantes, dependencias, decisiones)





02.30_SCOPE_AND_EXCLUSIONS (P0)


- Scope (qué se puede tocar):


  - PROJECT_ROOT por defecto


  - allowlist/denylist si aplica


- Exclusiones RADAR (P0):


  - RADAR_LITE y RADAR_CORE NO deben revisar:


    - <PROJECT_ROOT>\Raw\


    - <PROJECT_ROOT>\03_ARTIFACTS\ (incluye LOGS, RADAR\old, DeadHistory)


  - Implementación práctica recomendada:


    - excluir \Raw\ y \03_ARTIFACTS\ completo para CORE/LITE





02.40_DECISIONS


Por cada decisión:


- ID (WBS/único)


- Qué se decidió (texto corto)


- Por qué (razón)


- Estado:


  - DRAFT (aún discutible)


  - CANON (cerrado oficialmente)


- Evidencia:


  - link a archivo, commit, o sección





02.50_CHANGES_APPLIED


- Lista de cambios aplicados en filesystem:


  - operación (CREATE/EDIT/MOVE/RENAME/DELETE)


  - paths


  - commit id (si aplica)


  - quarantine path (si aplica)



02.80_NEXT_ACTIONS

02.60_VALIDATION_EVIDENCE


- OK/FAIL + evidencia:


  - logs


  - outputs esperados


  - comandos ejecutados


- Sin evidencia no hay “OK”.




02.65_COST_TRACKING (P0)


- Objetivo: que “control de costos” sea real (no PowerPoint).
- Siempre registrar:
  - provider/model (si aplica)
  - tipo de consumo: FIXED (suscripción) vs VARIABLE (API)
  - costo_estimado_usd (si corresponde)
  - costo_real_usd (si existe evidencia)
  - acumulado_periodo_usd (semana/mes)
- Alertas (MVP):
  - cada USD 30: registrar “ALERTA_30” + acumulado
    - ejemplo: “ALERTA_30: se cargaron 30 USD; acumulado=61 USD”
- Hard-stop (si aplica en este proyecto):
  - si acumulado_periodo_usd + costo_estimado_tarea > budget_periodo:
    - NO ejecutar APPLY
    - permitir PLAN_ONLY / dry-run
    - registrar decisión humana si se overridea (quién/cuándo/por qué)
- Backlog permitido:
  - si no se implementa logging automático hoy, igual se registra manual en BATON.




02.70_RISKS_AND_ASSUMPTIONS


- Riesgos abiertos (top 5–10)


- Supuestos activos (lo que se asumió por falta de datos)


- Plan para validar supuestos (cómo se obtiene dato duro)

- Lista WBS:


  - prioridad (P0/P1/P2)


  - owner (humano/IA planner/IA executor)


  - prerequisitos


  - definición de terminado de cada acción





02.90_HANDOFF_PACKAGE (por ContextLevel)


- Al migrar chat/IA:


  - Siempre:


    - BATON actualizado


    - Último commit id (si existe)


    - RADAR_INDEX + RADAR_LITE recientes (si existen)


  - Solo si la tarea lo exige (no por defecto):


    - RADAR_CORE por APP_SCOPE o módulo


  - Evitar:


    - pegar Raw / logs / artifacts completos


    - re-inyectar FULL sin razón (ruido y costo)

04.10_BATON_CON_GIT


- Git provee rollback.


- BATON apunta a commits como evidencia.





04.20_BATON_CON_RADAR


- RADAR_INDEX da mapa del repo.


- RADAR_LITE da delta de cambios.


- RADAR_CORE alimenta IA para comprensión del sistema (si se requiere y por scope).





04.30_BATON_CON_CIS

==========


05.00_MIGRACION_DE_CHAT_IA_EN_BLANCO


==========





05.10_SENALES_DE_DRIFT


- repetir contexto varias veces


- outputs inconsistentes


- confusión de paths o reglas


- “olvidos” operativos dentro de la sesión


- costo/tiempo sube sin progreso





05.20_PROCEDIMIENTO_DE_MIGRACION


Paso 1: Actualizar BATON (estado completo)


Paso 2: Ejecutar RADAR_INDEX + RADAR_LITE (foto + delta)


Paso 3: Si es necesario, RADAR_CORE por APP_SCOPE o módulo


Paso 4: Crear snapshot/export (si se requiere)


Paso 5: Abrir nuevo chat y “inyectar” solo:


        - BATON


        - RADAR_INDEX (+ LITE)


        - CORE solo si la tarea lo exige





05.30_QUE_EVITAR


- no pegar “todo el repo” sin necesidad


- no re-inyectar FULL como default


- no duplicar logs y basura (mantener relevancia)


- no incluir Raw / artifacts en CORE/LITE (P0)





==========


06.00_DOD_DE_SESION_(DEFINITION_OF_DONE)


==========





Sesión “DONE” si:


- BATON actualizado (02.00 completo)


- Evidencia mínima:


  - VALIDATE OK o FAIL explícito con logs


- Next actions claras (WBS + prioridad + owner)


- Si hubo APPLY:


  - commit id registrado


  - RADAR_LITE pre/post registrado (si aplica)
