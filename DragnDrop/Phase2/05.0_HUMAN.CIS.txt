00.10_GLOSARIO_MINIMO_(ABREVIACIONES_EXPLICADAS)


==========





CIS (Content Injection System): sistema de contratos para declarar cambios de forma inequívoca y ejecutable.


FS (Filesystem): sistema de archivos (carpetas/archivos) local del proyecto.


Planner (Planificador): rol (IA o humano) que redacta el contrato CIS; NO aplica cambios por defecto.


Executor (Ejecutor): rol (IA o humano) que aplica el contrato CIS bajo Gate.


Gate (Aprobación humana): permiso explícito para pasar de PLAN_ONLY a APPLY.


PLAN_ONLY (Solo plan): modo por defecto; produce contratos, no aplica.


APPLY (Aplicar): ejecución de cambios en FS; solo con gate humano.


VALIDATE (Validación): pruebas mínimas + verificación outputs.


LOG (Registro/Evidencia): log reproducible de OK/FAIL + cambios aplicados + referencias.


Quarantine (Cuarentena): destino por defecto para “deletes” (no hard delete).


CANON (Cierre oficial): estado congelado; cambios por hotfix/branch, no edición directa.


SYNC (Sincronización): replicación automática de bloques DERIVED; caso especial de cambios CIS.


DERIVED (Derivado): contenido generado/reemplazado por sync; no se edita manualmente.


Atomic Apply (Aplicación atómica): un target se escribe una sola vez, o no se escribe (evita APPLY parcial).

01.00_PROPOSITO_Y_MENTAL_MODEL


==========





01.10_PROPOSITO


- CIS existe para resolver un problema real:


  cuando un humano y una IA colaboran, el riesgo es drift, sorpresas y cambios invisibles.


- CIS fuerza que TODO cambio en FS sea:


  - explícito,


  - revisable,


  - aprobable (gate),


  - ejecutable,


  - reversible (rollback),


  - auditable (evidencia).





01.20_MENTAL_MODEL


- CIS es un contrato. Un contrato contiene:


  - alcance (scope),


  - operaciones,


  - contenido exacto (si aplica),


  - validación,


  - evidencia,


  - rollback.


- Si el contrato es ambiguo, CIS falla a propósito (fail-fast).


- Si el executor aplica algo no declarado, eso es “breach of contract” (incidente).





01.30_LO_QUE_CIS_NO_ES


- CIS NO es “IA haciendo magia”.


- CIS NO decide intención.


- CIS NO reemplaza Git ni RADAR.


- CIS NO reemplaza VALIDATE.





==========


02.00_ROLES_Y_SEPARACION_OBLIGATORIA


==========





02.10_PLANNER (PLANIFICADOR)


Responsabilidades:


- redacta el contrato CIS con precisión.


- declara supuestos y riesgos.


- propone scope y lista exacta de archivos a tocar.


- define validación mínima.


Prohibición por defecto:


- no aplica cambios (PLAN_ONLY).





02.20_EXECUTOR (EJECUTOR)


Responsabilidades:


- aplica el contrato exactamente como está escrito.


- no inventa pasos.


- ejecuta preflight y postflight.


- registra evidencia.


Requisito:


- gate humano explícito (APPLY_APPROVED).





02.30_POR_QUE_SEPARAR


- Sin separación se mezcla:


  - decisión con ejecución,


  - y se destruye la auditoría: no sabes qué fue “planeado” vs “aplicado”.





==========


03.00_GATES_Y_ESTADOS
==========


03.10_PLAN_ONLY (DEFAULT)


- Se permite:


  - diseñar contratos,


  - generar propuestas,


  - preparar scripts/patches.


- No se permite:


  - modificar FS,


  - borrar,


  - mover,


  - renombrar.





03.20_APPLY_APPROVED (SOLO CON GATE HUMANO)


Requisitos:


- contrato completo,


- scope confirmado,


- rollback Git listo,


- RADAR_LITE pre-change,


- validación ejecutable,


- secrets policy OK.





03.30_ESTADOS_DE_EJECUCION (RUN STATES)


- READY: contrato completo y aprobado.


- RUNNING: ejecución en curso.


- OK: ejecución + validación mínima OK.


- FAIL: ejecución o validación falla; rollback requerido.

03.40 REGLA P0: APPLY no es iteración eterna
- Si el contrato CIS ya cumple su OBJETIVO y VALIDACION pasa:
  - se declara OK,
  - se registra evidencia,
  - se cierra la MiniBattle.
- Mejoras adicionales:
  - se mueven a backlog (nuevo CIS_TASK_ID), no se “apilan” en el mismo APPLY.

04.00_OPERACIONES_CIS


==========





04.10_CREATE (CREAR)


- Crear archivo/carpeta nueva.


- Debe incluir:


  - path exacto,


  - contenido inicial si es archivo,


  - metadatos mínimos si es canónico.





04.20_EDIT (EDITAR)


- Para documentos HUMAN canónicos: preferir replace por bloque:


  - BLOQUE_ORIGINAL completo


  - BLOQUE_CORREGIDO completo


- Para código:


  - patch/diff preferido si pequeño,


  - archivo completo si es boilerplate o refactor masivo.





04.30_MOVE (MOVER)


- Origen → destino.


- Debe incluir impacto esperado (refs, manifests, scripts).





04.40_RENAME (RENOMBRAR)


- Igual que MOVE, pero con foco en:


  - naming,


  - IDs,


  - referencias cruzadas.





04.50_DELETE (ELIMINAR) — POR DEFECTO QUARANTINE


- Default: mover a quarantine/DeadHistory con timestamp y metadata.


- Hard delete: solo con orden humana explícita + checkpoint previo.

05.00_FORMATO_CANONICO_DEL_CONTRATO_CIS


==========





05.10_CONTRATO_CIS (PLANTILLA MINIMA)


Campos obligatorios:





- CIS_TASK_ID (ID único, WBS-friendly)


- TIPO (DOC / TOOL / CODE / AUDIT)


- OBJETIVO (qué cambia en el mundo)


- MODO (PLAN_ONLY / APPLY_APPROVED)


- SCOPE (paths permitidos + denylist)


- IMPACTO (qué archivos se tocarán y por qué)


- OPERACIONES (CREATE/EDIT/MOVE/RENAME/DELETE)


- VALIDACION (comandos/checks)


- EVIDENCIA (logs/commits/radar)


- ROLLBACK (plan Git / snapshot)


- RIESGOS (top 1–3)


- SUPUESTOS (si falta dato duro)





05.20_REGLA_DE_COMPLETITUD


- Si falta un campo, el contrato NO es ejecutable (fail-fast).





==========


06.00_REGLAS_DE_EJECUCION


==========





06.10_PREFLIGHT (ANTES DE APPLY)


Checklist mínimo:


- Gate humano confirmado (APPLY_APPROVED).


- Scope confirmado (no tocar fuera del root).


- Git checkpoint creado si habrá cambios.


- RADAR_LITE pre-change ejecutado (delta base).


- Validación definida y ejecutable.


- Secrets handling confirmado.





06.20_APPLY (EJECUCION)


Reglas:


- Ejecutar en orden declarado.


- Prohibido inventar pasos o tocar fuera de scope.


- Si una operación depende de otra (ej. rename + update refs), debe estar declarada como dos operaciones explícitas.





06.25_REGLA_DE_ATOMICIDAD_(P0)


- Si un runner aplica cambios “por target file” (ej. SYNC o batch edits):


  - debe validar TODO antes de escribir,


  - y escribir 1 vez por target (Atomic Apply).


- Motivo:


  - evita APPLY parcial (te pasó: aplicó 2 bloques y falló en el 3°, dejando el repo en estado híbrido).





06.30_POSTFLIGHT (DESPUES DE APPLY)


- RADAR_LITE post-change ejecutado.


- VALIDATE ejecutado.


- LOG generado con OK/FAIL y evidencia.


- BATON actualizado si el cambio es significativo.


- Commit Git trazable si OK.





06.40_FAIL_HANDLING


- Si FAIL:


  - detener cambios adicionales,


  - registrar evidencia,


  - rollback (Git),


  - registrar TROUBLE/INCIDENT según severidad.

07.00_EVIDENCIA_OBLIGATORIA


==========





07.10_RADAR


- RADAR_LITE pre/post APPLY obligatorio.


- RADAR_INDEX/CORE según necesidad:


  - CORE solo por APP_SCOPE o módulo (no Raw/Artifacts por política RADAR).





07.20_GIT


- Checkpoint antes.


- Commit después si OK.


- Branch para hotfix (CANON).





07.30_LOGS


- Logs reproducibles:


  - comandos,


  - outputs,


  - timestamps,


  - OK/FAIL.

08.20_HARD_DELETE_EXCEPCIONAL

==========


09.00_CIS_PARA_DOCUMENTACION_Y_SYNC


==========





09.10_SYNC_COMO_CASO_ESPECIAL_CIS


- SYNC es CIS aplicado a documentación:


  - toma fuente canónica,


  - reemplaza bloques DERIVED en destinos.


- Reglas:


  - solo reemplaza entre DERIVED_BEGIN / DERIVED_END.


  - MODE=FULL = copia completa (no resumen).


  - MODE=OVERRIDE bloquea reemplazo y marca DIRTY.





09.20_MARCADORES_DERIVED (ESTANDAR)


Inicio:


<<<DERIVED_BEGIN ID=<DERIVED_BLOCK_ID> SOURCE=<SOURCE_DOC_ID>::<SOURCE_WBS> MODE=FULL>>>


Fin:


<<<DERIVED_END ID=<DERIVED_BLOCK_ID>>>





09.30_DIRTY_DETECTION


- Si alguien editó manualmente dentro del bloque:


  - el runner debe detectarlo o al menos loguear “replaced regardless”.


- Política recomendada:


  - preferir reemplazo desde fuente (single-edit canon) y registrar el evento.
