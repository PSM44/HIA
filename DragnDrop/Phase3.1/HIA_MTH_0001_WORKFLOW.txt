01.00_OBJETIVO_DEL_METODO
==========

01.10_OPTIMIZA_POR
- Estabilidad del sistema (mínimo drift).
- Auditabilidad (evidencia siempre).
- Velocidad (menos rework por prompts malos).
- Control de costos (preflight + hard stop).

01.20_NO_OPTIMIZA_POR
- “Magia” sin controles.
- Automatización sin gates.
- Multiplicar herramientas/IDE/plugins (Frankenstein).

==========
03.00_CICLO_OPERATIVO_(PLAN_APPLY_VALIDATE_LOG)
==========

03.10_PLAN_(OBLIGATORIO)
Input mínimo:
- objetivo
- scope (paths o módulos)
- constraints (no tocar X)
Output mínimo:
- plan paso a paso
- riesgos y supuestos
- estimación costo (preflight)
- propuesta de patch (si aplica)

03.20_APPLY_(CON_GATE_HUMANO)
- Apply solo si:
  - plan aprobado,
  - sandbox activo,
  - checkpoint creado (Git o snapshot),
  - preflight dentro de budget.

03.30_VALIDATE_(MINIMO)
- Validación mínima:
  - outputs existen (si aplica),
  - smoke test,
  - script OK/FAIL real (si se ejecutó tooling).

03.40_LOG_(OBLIGATORIO)
- OK: registrar evidencia mínima.
- FAIL: crear TROUBLE_LOG con evidencia reproducible + rollback.

06.00_CANON_DEFINICION_OPERATIVA
==========

06.10_DEFINICION
Un archivo es CANON cuando:
(1) Cumple FILE_CONTENT al 100%.
(2) No contradice HUMAN root.
(3) Pasa validadores allowlisted.
(4) Tiene VERSION: vX.Y-CANON
(5) Tiene CHANGELOG completo.

06.20_REGLA
- IA no promueve a CANON.
- Humano promueve.

==========
07.00_CANON_PROMOCION_CONGELAMIENTO_Y_HOTFIX
==========

07.10_PROMOCION_(REVIEW_TO_CANON)
Pasos:
(1) Ejecutar validadores (naming + file_content).
(2) Registrar decisión en HIA_DEC_0001_DECISIONS.txt
(3) Cambiar VERSION a vX.Y-CANON
(4) Commit Git: "CANON: <ID_FS> vX.Y"

07.20_CONGELAMIENTO
- CANON no se edita directo en main.
- Cambios a CANON: branch + hotfix.

07.30_HOTFIX
(1) Crear branch: hotfix/<ID_FS>/<YYYYMMDD>
(2) Aplicar cambio mínimo (bloque).
(3) Subir versión: vX.(Y+1)-CANON
(4) Commit: "HOTFIX CANON: <ID_FS> vX.(Y+1)"
(5) Merge.
