00.30_DEFINICIONES_BASE_(ABREVIACIONES_EXPLICADAS)
==========

HIA (Human + IA): entorno/metodología reusable para colaborar humano + IA.
TL.Bootstrap (Tooling & Lifecycle Bootstrap): set transversal reusable (RADAR, CIS, BATON, PF.0, estructura, tools).
Cambio: crear / editar / renombrar / mover / eliminar (archivo o carpeta).
Gate: punto de control humano para pasar de PLAN a APPLY.
PLAN_ONLY: solo planificación, sin modificar FS.
APPLY_APPROVED: ejecución permitida por gate explícito.
DoD (Definition of Done): criterio de terminado (por fase / por minibattle).
CANON: cierre oficial; congelado; no editar directo en main.
MB (MiniBattle): unidad mínima de avance verificable.
FS: filesystem del proyecto.

01.00_FILOSOFIA_Y_PRINCIPIOS
==========

01.10_NO_NEGOCIABLES
- HUMAN manda.
- PLAN→APPLY→VALIDATE→LOG (siempre).
- Git siempre.
- Delete=quarantine (por defecto).
- No magia: supuestos explícitos y plan de validación.

01.20_ANTI_OBJETIVOS
- Automatización total sin gates.
- “Inyectar todo siempre”.
- Tooling por ansiedad (Frankenstein).
- Reemplazos totales cuando bastan bloques.

01.30_PIRAMIDE_DE_AUTORIDAD
- Nivel 0: HUMAN (conceptos/decisiones).
- Nivel 1: Políticas (gates, canon, seguridad).
- Nivel 2: Métodos (workflow, routing).
- Nivel 3: Tools (scripts/validadores).
- Nivel 4: Código de app.
- Nivel 5: Artefactos (logs/snapshots).

01.40_EVIDENCIA
- Decisión importante = qué / por qué / cuándo / quién + evidencia mínima.

02.00_GOBERNANZA_OPERACIONAL
==========

02.10_ROLES
- HUMANO: intención, prioridades, restricciones, gates, promoción a CANON.
- IA_PLANNER: propone plan/contrato, default PLAN_ONLY.
- IA_EXECUTOR: aplica cambios solo con gate + CIS + evidencia.

02.20_GATES
- Default: PLAN_ONLY.
- APPLY_APPROVED solo con:
  - contrato completo,
  - rollback Git,
  - RADAR_LITE pre/post,
  - validate ejecutable.
- Regla peatón (P0): gates pocos y claros.
  - Gate humano máximo por acción: 1 (PLAN->APPLY). Evitar “doble aprobación” por costumbre.
  - Adjuntos por defecto: 0.
  - Adjuntos permitidos en MVP: 1 (solo si desbloquea la tarea).
  - Si se necesita >1 adjunto o contexto “pesado”:
    - escalar por excepción (registrar en BATON: razón + costo estimado + DoD).

02.30_CICLO_OPERATIVO
- PLAN: objetivo + scope + constraints + output + validación.
- APPLY: ejecutar exactamente lo planeado.
- VALIDATE: pruebas mínimas.
- LOG: evidencia OK/FAIL + cambios + next actions.

02.40_CONTEXT_LEVELS
- L0: HUMAN/BATON + navegación básica.
- L1: snippets selectivos.
- L2: RADAR_CORE por módulo/APP_SCOPE.
- L3: RADAR_FULL puntual (solo forensics excepcionales autorizados; no es output estándar).

02.41_POLITICA_COSTO_Y_PERFORMANCE_(NO_POWERPOINT)
Regla:
- “Control de costos” existe solo si hay:
  - métrica,
  - umbral,
  - alerta,
  - y registro (para estimar costos futuros).

02.41.10_ORDEN_DE_IMPLEMENTACION_(REALISTA)
1) Benchmark local (si aplica):
   - correr herramienta tipo llmfit (o equivalente) para determinar modelos locales viables según hardware.
   - output mínimo: tabla “modelo / tokens/s / RAM / calidad aproximada / tarea recomendada”.
2) Inventario de herramientas pagadas:
   - costo fijo (ej: ChatGPT Plus),
   - costo variable (APIs / Claude Code / etc.).
3) Umbrales operativos (alertas):
   - ejemplo: cada +USD 30 acumulados => alertar “acumulado = X”.
   - (si no hay integración automática, igual se registra manual en LOG/BATON).
4) Registro para forecast:
   - por MiniBattle: input_size, context_level usado, modelo, duración, costo estimado/real, outcome.

02.41.20_ARTEFACTO_MINIMO_(BACKLOG_READY)
- COST.TRACKER (puede vivir en BATON o en un archivo dedicado del proyecto):
  - MB_ID
  - tool/model
  - context_level
  - costo_usd (estimado/real)
  - acumulado_usd
  - timestamp
  - nota

02.42_STACK_SIN_RELIGION_(ITERAR_POR_CASO)
- Stack no se “elige para siempre” en F0.
- Se propone una hipótesis por caso, se valida con smoke, y se registra decisión DRAFT/CANON.
- Si no se puede implementar completo ahora:
  - queda explícito en backlog (no se “olvida”).

05.00_RADAR
